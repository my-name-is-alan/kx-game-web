# 动态商城购买输入与余额提示设计

## 目标

改善动态商城购买弹窗在大购买上限下的数量控制，并在用户点击购买前明确提示货币不足。

## 交互设计

- 保留现有滑杆与 `-1`、`+1` 微调。
- 将数量展示改为可点击的正整数输入，用户可直接输入精确数量。
- 增加“最大”快捷操作，一次选择本商品当前允许购买的最大数量。
- 数量输入、滑杆、加减和最大操作共用同一套数量归一化逻辑，范围始终限制在 `1..maxCount`。
- 数量变化后继续请求服务端实时报价，报价返回前显示“报价中...”并禁用购买。
- 报价成功后，将报价涉及的玉璧、灵石、代金券分别与玩家当前余额比较。
- 任一货币不足时，报价和“货币不足”显示红色并禁用购买；全部足够时恢复正常颜色并允许购买。
- 最终购买仍由服务端再次校验，客户端提示只用于提前反馈，不替代服务端权威校验。

## 实现边界

- 只修改动态商城购买弹窗，不改变普通活动商城购买逻辑。
- 不修改服务端报价、购买协议或 GM 配置结构。
- 不改动现有无关工作区文件。

## 验收标准

- 大上限商品可以直接输入任意合法购买数量，也可以一键选择最大数量。
- 输入空值、非数字、零、负数或超过上限时会被安全归一化，不发送非法数量。
- 快速连续修改数量时，旧报价响应不能覆盖新数量的报价。
- 余额不足无需点击购买即可看到红色提示，购买按钮不可用。
- 余额足够时报价正常显示，购买按钮可用，购买流程保持现有行为。
