# 坊市货币图标加载崩溃修复设计

## 背景与根因

`1.0.16` 将坊市价格改为动态 `GRichTextField`，通过 `[img]ui://CCommon/...[/img]` 嵌入货币图标。FairyGUI 的富文本图片解析会在带遮罩的组件构造期间同步读取尚未完成懒加载的 `CCommon` 图集；当 `CCommon_atlas0` 或 `CCommon_atlas10` 尚不可用时，后续遮罩初始化收到空渲染对象并在设置 `stencilStage` 时崩溃。

项目现有的稳定资源加载方式是把 `UtilsUI.getItemIconUrl(...)` 赋给 `GLoader.url`，由 FairyGUI 已有的异步图集更新路径完成占位图到真实图集的切换。

## 目标与范围

- 坊市商品列表和购买弹窗继续显示灵石、玉璧、代金券的现成图标。
- 完全移除坊市价格路径中的动态 `GRichTextField` 和 `[img]` 图片标签。
- 保留数量输入、最大数量按钮、服务端报价、余额不足红色提示和购买按钮禁用逻辑。
- 不修改服务端报价协议、商城配置、其他界面或全局 FairyGUI 资源策略。

## 方案

新增一个只负责渲染报价的本地辅助逻辑，把报价规范化为最多三个有序货币项：灵石、玉璧、代金券。每一项由一个 `GLoader` 和一个 `GTextField` 组成；金额为零的项隐藏，所有项均为零时显示“免费”。

商品列表优先复用原有价格图标和文本控件，不足的槽位按同一 `GLoader + GTextField` 结构创建。购买弹窗使用同样的槽位渲染方式。每次服务端报价返回时，只更新槽位的 `url`、金额文本、位置和可见性，不创建富文本图片。

余额判断继续使用现有 `getBazaarQuoteAffordability`。余额不足时，金额文本与“货币不足”提示统一使用现有不足颜色，购买按钮保持禁用；资源加载失败不得改变报价和购买状态。

## 数据流

1. 客户端向服务端请求指定购买数量的报价。
2. 将 `moneyCost`、`stoneCost`、`voucherCost` 转成有序货币项。
3. 用 `UtilsUI.getItemIconUrl(type)` 设置每个 `GLoader.url`，用普通文本字段显示金额。
4. 根据本地余额设置正常色或不足色，并决定购买按钮是否可用。

## 测试与验收

- 合同测试必须确认坊市实现不再包含 `GRichTextField`、`ubbEnabled` 或 `[img]` 价格图片路径。
- 合同测试必须确认列表和购买弹窗都使用 `GLoader` 与 `UtilsUI.getItemIconUrl`。
- 既有数量规范化、余额不足和邮件竞态测试继续通过。
- Cocos Web 构建必须成功，生成包中不得再出现坊市价格富文本图片实现。
- 浏览器验收由用户执行：打开商城和购买弹窗时不再出现 `CCommon_atlas0/10` 与 `stencilStage` 错误；图标、金额、红色不足提示和数量输入均正常。
