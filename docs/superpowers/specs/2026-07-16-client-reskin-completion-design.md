# 客户端换皮遗留收口设计

## 目标

修复仍会直接影响玩家的侠侣获取渠道、精英等级判断和侠侣说明文案，并让客户端生成数据严格跟随服务端共享配置。

## 获取渠道

- 混元散 120012 和九仪珀 120013 的 `getParam="2,11"` 必须同时存在于客户端源 XML 与生成 JSON。
- `LyGuideGetItem` 继续使用现有 2=坊市、11=魔教来袭路由；契约测试要验证两种入口都会渲染，且坊市能匹配新增商品。
- 不在视图里硬编码材料名称或活动跳转。

## 精英共鸣等级

- `LyEliteGroup` 必须先从服务端数据读取 `resonanceLevel`，再判断是否达到配置上限。
- 两处初始化/刷新路径使用同一计算顺序，避免初始 `0` 导致满级状态错误。

## 侠侣说明文案

- 修复 Pet 源表中重复编号、缺少换行以及“羁绊六、洗练”粘连。
- 文案从 `Excel/Pet-侠侣（宠物）.xls` 生成到服务端 Pet.xml，再同步到客户端 Pet.xml/Pet.json；禁止只改 JSON。

## 旧协议兼容

- 客户端没有已完成的协战侠侣界面和消息处理，因此不新增入口。
- 保留生成协议中的 `assignPet` 字段和消息编号，避免改变后续字段/消息布局；运行层不注册功能。

## 共享配置

- Activities、CombatPower、Evolution、Items、Task 由服务端生成结果同步。
- 客户端 JSON 通过现有 `gen_script.lua xmltojson` 生成。
- 前后端 XML 语义一致，XML/JSON 关键字段一致。

## 验证

- 获取途径面板能列出坊市与魔教来袭，且两个材料均可匹配。
- 满级和未满级精英共鸣状态用测试夹具覆盖。
- Pet 三份说明字段编号连续、分段完整、不含粘连文本。
- 所有修改后的 JSON 可解析，TypeScript 静态检查和现有契约测试通过。
